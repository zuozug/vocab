# 单词本项目需求文档

## 1. 项目概述

本项目旨在开发一个简单、可长期使用的单词本系统，核心目标是帮助用户以较低维护成本手动积累词汇，并通过主动回忆的方式进行背诵与复习。项目需要支持多设备使用，能够在电脑与手机之间同步数据，并具备基础的学习记录与历史查询能力。

项目定位以“个人使用、功能实用、实现清晰”为主，不追求复杂的社交、推荐或自动导词功能，优先保证录入、背诵、记忆安排和历史统计这几项核心能力可用。

---

## 2. 项目目标

### 2.1 核心目标

- 支持用户手动维护自己的词库。
- 支持围绕“释义回忆单词”展开的主动背诵。
- 支持中英双向复习。
- 提供简单可实现的记忆调度逻辑，而非复杂算法。
- 记录每次学习情况，并支持查看历史记录。
- 支持手机与电脑访问同一套数据。

### 2.2 非目标

当前阶段暂不作为必须需求考虑以下内容：

- OCR 扫描导词
- 拍照识别单词
- 联网词典自动补全释义
- 社交功能
- 排行榜
- 多人协作
- 复杂 AI 出题
- 过于复杂的记忆曲线模型

---

## 3. 用户场景

### 3.1 单词录入场景

用户在日常学习中遇到一个新单词，希望手动将其录入单词本。录入内容至少包括：

- 单词本身
- 词性
- 中文释义

同一个单词可能包含多个词性，也可能在同一词性下有多个中文释义，因此系统不能把一个单词限制为单一词性或单一释义。

### 3.2 背诵场景

用户不希望通过“认识 / 不认识”或单纯选择题来背单词，而更倾向于主动输入答案。系统给出提示后，用户输入自己的答案，系统对答案进行匹配和判断。

背诵应至少支持两种模式：

1. 给出中文释义，用户输入英文单词。
2. 给出英文单词，用户输入中文释义。

### 3.3 复习调度场景

用户希望系统能根据记忆情况安排复习，但不要求复杂算法。系统只需根据单词熟练度和上次复习时间，安排较简单的优先复习逻辑即可。

### 3.4 历史查询场景

用户希望每次学习结束后，系统记录本次学习的基础信息，例如：

- 本次背诵的单词数量
- 本次学习耗时
- 学习时间

用户后续可以查看这些历史记录，以了解自己的学习情况。

### 3.5 多设备使用场景

用户可能同时在电脑和手机上使用该系统，因此数据需要跨设备同步。项目后续会部署到服务器，并连接在线数据库。

## 4. 功能需求

## 4.1 单词管理

### 4.1.1 新增单词

系统应支持用户手动新增单词。

每个单词至少包含以下字段：

- 单词拼写
- 一个或多个词性
- 每个词性下的一个或多个中文释义

当前数据库将词性限制为固定枚举，初版支持：

- `n`
- `v`
- `adj`
- `adv`
- `prep`
- `pron`
- `conj`
- `interj`
- `num`
- `art`

### 4.1.2 单词结构要求

一个单词需要支持如下层级关系：

- 一个单词
  - 多个词性
    - 每个词性下多个中文释义

示例：

- `record`
  - `n.`：记录；唱片
  - `v.`：记录；录制

### 4.1.3 编辑单词

系统应支持对已录入单词进行修改，包括：

- 修改单词拼写
- 增加或删除词性
- 修改词性内容
- 增加或删除中文释义
- 修改中文释义

### 4.1.4 删除单词

系统应支持删除单词。

删除后，该单词及其词性、释义、复习状态不再参与后续背诵。当前数据表采用外键约束，应用层删除时需要先处理关联的 `meaning`、`word_sense`、`review_state` 数据，再删除 `word`。

### 4.1.5 查询单词

系统应支持按关键词查询单词，至少支持：

- 按英文单词查询
- 查看单词详情

后续可扩展：

- 按中文释义搜索
- 按词性过滤

---

## 4.2 背诵功能

### 4.2.1 背诵模式

系统至少支持以下两种背诵模式：

#### 模式 A：中文到英文

- 系统展示中文释义
- 用户输入对应英文单词
- 系统只判断英文拼写是否正确

#### 模式 B：英文到中文

- 系统展示英文单词
- 用户输入词性和中文释义
- 系统需要判断所有词性下的所有释义是否全部匹配，顺序不重要

### 4.2.2 输入作答

系统不应只提供“认识 / 不认识”按钮作为唯一作答方式，而应支持用户主动输入答案。

这项需求是本项目区别于简单打卡式背单词工具的重要点。

### 4.2.3 答案判断

系统需要对用户输入进行比对。

基础要求：

- 支持与标准答案进行字符串匹配
- 对英文答案可考虑基础的大小写宽容处理
- 中文到英文时，用户只要拼对 `word.spelling` 即可。
- 英文到中文时，用户输入只包含词性、释义和空格；每个释义归属于它之前最近一次出现的词性。
- 英文到中文必须匹配该单词所有 `(词性, 释义)` 项，不能少也不能多；顺序不重要。
- 例如输入 `n 记录 v 记录 n 猫 狗`，表示三个 `n` 释义：`记录`、`猫`、`狗`，以及一个 `v` 释义：`记录`。
- 对中文释义可支持与标准释义之一匹配即算正确

当前阶段可以采用相对简单的判断逻辑，不要求复杂自然语言理解。

### 4.2.4 结果反馈

每次作答后，系统应至少反馈：

- 是否正确
- 正确答案是什么

后续可扩展：

- 展示该单词的全部词性与释义
- 标记本次是否为首次记对

---

## 4.3 记忆与复习调度

### 4.3.1 调度原则

系统需要具备基本的记忆调度能力，但应保持实现简单。

用户倾向于采用如下思路：

- 为每个单词记录一个整数熟练度
- 记录单词上一次复习时间
- 复习时以熟练度为第一权重，上次复习时间为第二权重排序

### 4.3.2 最低记录项

每个单词至少应记录与记忆状态相关的信息，例如：

- 当前熟练度
- 上次复习时间

熟练度规则：

- 熟练度为整数。
- 熟练度最小值为 `0`。
- 新单词初始熟练度为 `0`。
- 单词在一段时间内未复习时，系统应自动降低熟练度。
- 新单词创建复习状态时，`last_reviewed_at` 设为当前时间。
- V1 默认规则：距离 `last_reviewed_at` 每满 `1` 天未复习，`proficiency - 1`，最低保持为 `0`。
- 未复习衰减只作用于 `proficiency > 0` 的单词。
- 系统应记录最近一次应用衰减的时间，避免同一段未复习时间被重复扣减。

### 4.3.3 初期实现建议

初版不要求引入复杂的间隔重复算法，可以先实现如下逻辑：

- 新单词熟练度为 `0`，优先进入待复习列表。
- 作答正确后，熟练度 `+1`。
- 作答错误后，熟练度 `-2`，但最低不能低于 `0`。
- 每次作答后更新 `last_reviewed_at`。
- 查询待复习单词前，先自动应用未复习熟练度衰减，避免长期未复习的高熟练度单词一直排在后面。
- 系统需要避免同一单词在短时间内被重复执行同一段未复习时间的衰减。
- 查询待复习单词时，优先选择熟练度低的单词；熟练度相同时，优先选择更久没有复习的单词。

后续如有需要，再演进为更完整的记忆算法。

---

## 4.4 学习记录与统计

### 4.4.1 学习会话记录

每次背诵结束后，系统应生成一条学习记录。

记录内容至少包括：

- 学习开始时间或结束时间
- 本次背诵单词数量
- 本次学习耗时

### 4.4.2 历史查询

系统应支持查看历史学习记录列表。

至少支持：

- 按时间倒序查看
- 查看某次学习的基础信息

后续可扩展：

- 按天汇总
- 按周汇总
- 统计累计背诵次数
- 统计正确率

---

## 5. 数据需求

## 5.1 单词数据

单词数据至少需要表达以下信息：

- 单词基础信息
- 多词性
- 多释义
- 记忆状态信息

### 当前采用的数据关系

- `word`
  - id
  - spelling
  - created_at
  - updated_at

- `word_sense`
  - id
  - word_id
  - part_of_speech
  - created_at
  - updated_at

- `meaning`
  - id
  - sense_id
  - definition
  - created_at
  - updated_at

- `review_state`
  - word_id
  - proficiency
  - last_reviewed_at
  - last_decay_applied_at
  - created_at
  - updated_at

当前约束：

- `word.spelling` 唯一。
- `word_sense` 中同一个单词下同一词性只能存在一条记录。
- `meaning` 中同一个词性下相同释义只能存在一条记录。
- `review_state` 与 `word` 一对一。
- `review_state.proficiency` 为非负整数，默认值为 `0`。
- `word`、`word_sense`、`meaning`、`review_state` 的 `updated_at` 由数据库触发器自动维护。

### 5.2 学习记录数据

- `study_session`
  - id
  - started_at
  - ended_at
  - reviewed_word_count
  - duration_seconds

当前 V1 只记录学习会话级别的数据，不单独记录每一道题的作答明细。

如后续需要更详细统计，也可以增加 `review_log`：

- `review_log`
  - id
  - session_id
  - word_id
  - prompt_type
  - user_answer
  - is_correct
  - reviewed_at

`review_log` 可用于支持正确率、错题本、单词级答题历史等功能，但不属于当前 SQL 文件已经敲定的 V1 数据表。

## 6. 非功能需求

## 6.1 多设备同步

由于用户希望手机和电脑互通，系统需要保证不同设备访问的是同一套后端数据。

因此项目应满足：

- 数据存储在统一的在线数据库中
- 客户端通过网络访问后端服务
- 不依赖单机本地数据库作为唯一数据源

## 6.2 可部署性

项目应可部署到个人服务器，运行在 Debian 环境中。

因此开发过程中应注意：

- 配置与代码分离
- 避免写死本地 Windows 路径
- 为开发环境和生产环境保留不同配置

## 6.3 易维护性

项目定位为个人长期使用工具，代码结构应尽量清晰，便于后续自己维护与迭代。

## 6.4 简洁优先

在功能设计上，应优先保证核心流程清楚可用，不以堆叠复杂功能为目标。

---

## 7. 建议的版本范围

## 7.1 V1 最小可用版本

V1 建议只实现以下能力：

1. 手动新增单词
2. 一个单词支持多个词性与多个中文释义
3. 支持中文到英文背诵
4. 支持英文到中文背诵
5. 支持输入作答与基础答案判断
6. 基于 `proficiency` 和 `last_reviewed_at` 记录简单记忆状态
7. 记录每次学习的数量与耗时
8. 支持历史记录查询
9. 支持多设备访问同一后端

## 7.2 V2 可扩展方向

后续可考虑加入：

- 中文释义搜索
- 标签或单词分组
- 错题本
- 每日学习目标
- 更完整的复习算法
- 导入导出功能
- 更细粒度的统计图表

---

## 8. 验收标准

若满足以下条件，可认为项目初版基本达到目标：

- 用户可以手动添加一个单词，并为其录入多个词性和多个释义。
- 用户可以查看、修改、删除已录入单词。
- 用户可以进行中文到英文、英文到中文两种背诵。
- 用户在背诵时需要主动输入答案，而不是只点按钮。
- 系统能根据 `review_state.proficiency` 和 `review_state.last_reviewed_at` 安排待复习单词。
- 每次学习结束后，系统能记录学习数量与耗时。
- 用户可以查看历史学习记录。
- 用户在不同设备上访问时，能够使用同一套数据。

---

## 9. 一句话总结

这是一个以“手动积累词汇 + 主动输入背诵 + 简单复习调度 + 多设备同步”为核心的个人单词本项目，初版重点不是功能堆砌，而是把单词管理、背诵、复习和学习记录这条主流程完整打通。
