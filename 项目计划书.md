# 单词本项目计划书

## 1. 项目定位

本项目是一个个人长期使用的单词本系统，核心目标是打通“手动录入单词 -> 主动输入背诵 -> 简单复习调度 -> 学习历史记录 -> 多设备同步”这条主流程。

当前技术栈暂定为：

- 前端：Vue 3 + Vite
- 后端：FastAPI
- 数据库：PostgreSQL

项目优先保证核心流程清晰可用，不在初版堆叠复杂功能。

---

## 2. 总体推进原则

### 2.1 后端优先

本项目的核心复杂度主要在数据结构、接口设计和复习状态更新。应先把 PostgreSQL schema、FastAPI 数据模型和 API 主流程稳定下来，再推进前端页面。

推荐顺序：

1. 数据库 schema 固化
2. 后端项目初始化
3. 后端接口实现与自测
4. 前端项目初始化
5. 前后端联调
6. 部署与移动端适配

### 2.2 先闭环，后优化

第一阶段不追求完整统计、错题本、导入导出、复杂复习算法。优先完成一个可以每天使用的版本。

V1 的最小闭环是：

1. 新增、查询、编辑、删除单词
2. 支持多词性、多释义
3. 支持中文到英文背诵
4. 支持英文到中文背诵
5. 主动输入答案并判断正确性
6. 更新 `review_state`
7. 记录 `study_session`
8. 查看历史学习记录

## 5. 阶段三：单词管理 API

### 5.1 目标

实现单词的新增、查询、详情、编辑、删除能力。

### 5.2 API 建议

```text
POST   /api/words
GET    /api/words
GET    /api/words/{word_id}
PUT    /api/words/{word_id}
DELETE /api/words/{word_id}
```

### 5.3 请求模型建议

新增或编辑单词时，前端提交完整结构：

```json
{
  "spelling": "record",
  "senses": [
    {
      "part_of_speech": "n",
      "meanings": ["记录", "唱片"]
    },
    {
      "part_of_speech": "v",
      "meanings": ["记录", "录制"]
    }
  ]
}
```

### 5.4 实现顺序

1. 实现单词详情查询。
2. 实现单词列表查询。
3. 实现新增单词。
4. 实现编辑单词。
5. 实现删除单词。

这个顺序可以先把数据读取结构稳定下来，再处理写入逻辑。

### 5.5 关键规则

- `spelling` 不能为空。
- 同一个单词下不能重复出现同一词性。
- 同一词性下不能重复出现相同中文释义。
- 新增单词成功后，应同步创建 `review_state`，或在首次进入复习队列时按需创建。推荐 V1 在新增单词时创建。
- 删除单词时，需要删除关联数据。

### 5.6 交付物

- 单词管理 API。
- 单词相关 Pydantic schema。
- 单词相关 service/repository。
- 基础接口测试。

### 5.7 完成标准

- 可以通过 API 新增 `record`，并录入 `n`、`v` 两个词性。
- 可以查询单词列表。
- 可以查看单词详情。

## 6. 阶段四：背诵与复习 API

### 6.1 目标

实现待复习单词获取、答案提交、答案判断和复习状态更新。

### 6.2 API 建议

```text
POST /api/study-sessions
GET  /api/reviews/next?mode=zh_to_en&limit=10
POST /api/reviews/answer
POST /api/study-sessions/{session_id}/finish
```

### 6.3 背诵模式

V1 支持两种模式：

- `zh_to_en`：展示中文释义，用户输入英文单词。
- `en_to_zh`：展示英文单词，用户输入中文释义。

### 6.4 待复习选择逻辑

V1 使用简单规则：

1. 获取待复习队列前，先自动应用未复习熟练度衰减。
2. 以 `review_state.proficiency` 为第一排序权重，熟练度越低越优先。
3. 以 `review_state.last_reviewed_at` 为第二排序权重，越久没有复习越优先。
4. 新单词创建时 `last_reviewed_at` 默认为当前时间；同熟练度下，仍优先选择更久没有复习的单词。
5. 如果待复习数量不足，可继续返回熟练度较高的单词。

### 6.5 答案判断规则

中文到英文：

- 用户答案去除首尾空格。
- 英文比较时忽略大小写。
- 与 `word.spelling` 相等则正确。

英文到中文：

- 用户答案去除首尾空格。
- 用户输入只包含词性、释义和空格；每个释义归属于它之前最近一次出现的词性。
- 必须匹配该单词所有 `(part_of_speech, meaning.definition)` 项，顺序不重要。
- 不能少写释义、不能多写释义，也不能写错词性。
- 例如 `n 记录 v 记录 n 猫 狗` 表示三个 `n` 释义：`记录`、`猫`、`狗`，以及一个 `v` 释义：`记录`。
- V1 暂不做同义词、模糊匹配或自然语言理解。

### 6.6 复习状态更新规则

答对：

- 更新 `proficiency = proficiency + 1`。
- 更新 `last_reviewed_at = now()`。

答错：

- 更新 `proficiency = greatest(proficiency - 2, 0)`。
- 更新 `last_reviewed_at = now()`。

未复习自动衰减：

- 当单词 `proficiency > 0`，且距离上次复习已满 `1` 天时，系统自动降低熟练度。
- V1 默认按每满 `1` 天降低 `1` 级计算，即 `proficiency = greatest(proficiency - elapsed_day_periods, 0)`。
- 衰减应在获取待复习队列前执行，保证排序使用的是衰减后的熟练度。
- 衰减逻辑通过 `review_state.last_decay_applied_at` 记录最近一次应用衰减的时间，避免对同一段未复习时间重复扣减。
- 新单词创建复习状态时，`last_reviewed_at` 设为当前时间。

### 6.7 学习会话规则

开始学习：

- 创建 `study_session`。
- 写入 `started_at`。

结束学习：

- 写入 `ended_at`。
- 写入 `reviewed_word_count`。
- 写入 `duration_seconds`。

### 6.8 交付物

- 复习队列 API。
- 答案提交 API。
- 复习状态更新逻辑。
- 未复习熟练度自动衰减逻辑。
- 学习会话开始和结束 API。

### 6.9 完成标准

- 可以创建一次学习会话。
- 可以获取待复习单词。
- 可以提交中文到英文答案。
- 可以提交英文到中文答案。
- 答案正确时熟练度 `+1`，并更新 `last_reviewed_at`。
- 答案错误时熟练度 `-2`，最低保持为 `0`，并更新 `last_reviewed_at`。
- 长时间未复习的单词会在进入待复习排序前自动降低熟练度。
- 学习结束后生成 `study_session` 记录。

## 17. V1 最终验收清单

- 可以新增一个单词。
- 一个单词可以包含多个词性。
- 一个词性可以包含多个中文释义。
- 可以查看、编辑、删除单词。
- 可以进行中文到英文背诵。
- 可以进行英文到中文背诵。
- 背诵必须由用户主动输入答案。
- 系统可以判断答案是否正确。
- 答题后可以展示正确答案。
- 答对后可以将 `proficiency` 增加 `1`。
- 答错后可以将 `proficiency` 减少 `2`，但不低于 `0`。
- 长时间未复习的单词可以自动降低 `proficiency`，且最低不低于 `0`。
- 系统可以按熟练度和上次复习时间安排复习顺序。
- 每次学习结束后可以生成 `study_session`。
- 可以查看历史学习记录。
- 电脑和手机可以访问同一套后端数据。
